# 04-规则引擎与工作流设计

## 1. 文档口径

本文档描述规则引擎和审核工作流的目标设计。当前内容为建议实现，不表示规则引擎代码已经完成。

## 2. 设计目标

规则引擎目标是回答 `NOTE.md` 中的几个核心问题：

- API 如何区分公司、业务、模态和策略。
- 规则引擎应硬编码还是可视化。
- 处理策略调用什么工具或算法。
- 处理尺度如何定义和调整。
- 输出仅违规结论还是详细标签。
- 前端和策略设计如何做到可演示、可操作。

建议实现以配置化规则为主，必要时保留少量底层硬编码安全约束。前端应提供可视化配置和策略预览，避免规则只能在代码中维护。

## 3. 规则模型

规则应按策略版本管理，核心结构沿用 `PolicyRule`：

- `rule_id`：规则唯一 ID。
- `label`：命中标签。
- `condition`：命中条件。
- `threshold`：阈值。
- `action`：处置动作。
- `enabled`：是否启用。
- `version`：规则版本。

规则生效范围建议包括：

- `tenant_id`：公司或租户。
- `business_id`：业务线。
- `modality`：内容模态。
- `policy_id`：策略。
- `version`：策略版本。

## 4. 条件类型

建议支持以下条件类型：

- 关键词命中：适用于敏感词、广告导流词等。
- 正则命中：适用于联系方式、外链、变体文本等。
- 模型分数阈值：适用于图片、文本、视频、音频模型标签分数。
- 标签组合：适用于多个弱风险组合后触发复核。
- Graph RAG 节点命中：适用于命中特定政策、案例或高风险知识节点。
- 业务上下文条件：适用于不同业务场景的审核尺度差异。

## 5. 阈值与处理尺度

处理尺度应通过阈值和策略版本表达：

- 宽松策略：高阈值触发复核或拒绝，适用于低风险业务。
- 标准策略：中等阈值触发复核，适用于通用业务。
- 严格策略：低阈值触发复核或拒绝，适用于未成年、政务、公开传播等高风险场景。

前端应支持调整阈值并预览同一条内容在不同阈值下的 `decision`、`risk_score`、`labels` 和 `suggested_action` 变化。

## 6. 处置策略

建议动作包括：

- `pass`：通过。
- `review`：进入人工复核。
- `reject`：拒绝或拦截。
- `pass_with_limit`：限制传播后通过。
- `escalate`：升级给更高权限审核员或专项模型。

处置策略应由规则结果、模型结果和 Graph RAG 证据共同决定。规则引擎不应只输出布尔值，而应输出命中规则、证据、标签、分数和建议动作。

## 7. 审核工作流

目标工作流：

1. 接收 `ModerationRequest`。
2. 校验租户、业务、模态和策略版本。
3. 执行模型路由，获得 `model_results`。
4. 执行规则引擎，获得 `rule_results`。
5. 调用 Graph RAG，获得政策、案例和规则证据。
6. 融合模型、规则和 RAG 结果，生成 `ModerationResult`。
7. 对 `review` 结果进入人工复核。
8. 人工复核可一键通过、打回、改标、备注。
9. 全流程写入 `AuditTrace`。

## 8. 可视化配置

前端可视化配置应支持：

- 规则列表：规则 ID、标签、状态、版本、最近修改人。
- 规则编辑：条件、阈值、动作、启用状态。
- 策略预览：调整阈值后展示结果变化。
- 命中解释：高亮证据片段，展示规则来源和模型来源。
- 版本对比：展示不同策略版本下规则差异。

## 9. 自动化流程

规则引擎和工作流应支持自动化：

- 新规则提交后自动运行回归样本测试。
- 策略上线前自动生成影响评估。
- 审核任务失败自动重试或降级。
- 高风险标签自动进入人工复核队列。
- 复核结果自动回流到训练和规则优化数据。

## 10. 输出粒度

后续实现应支持两种输出粒度：

- `basic`：输出 `decision`、`risk_score`、`suggested_action`。
- `detailed`：额外输出 `labels`、`evidence`、`model_results`、`rule_results`、Graph RAG 证据和解释。

## 11. 智能体触发规则引擎协议

智能体通过内部 API 触发规则引擎，默认使用 HTTP/JSON 同步调用：

- 查询规则：`POST /internal/v1/rules/query`。
- 执行规则：`POST /internal/v1/rules/evaluate`。

同步和异步边界：

- 单条文本、图片等轻量任务默认同步执行规则，便于 3 秒内返回前端可展示结果。
- 批量任务、视频任务、大量规则回归测试或规则执行失败重试进入任务队列异步处理。
- 异步执行必须返回 `task_id`、`status` 和 `trace_id`，并将最终结果写入 `AuditTrace`。

规则执行输入必须包含标准化 `model_results`。如果模型尚未返回结果，规则引擎只能执行关键词、正则、业务上下文等不依赖模型分数的规则；依赖 `model_score` 的规则应返回未满足输入条件的错误或等待异步补算。

## 12. RuleEvaluationRequest

规则执行请求沿用 `03-API协议设计.md` 中的 `RuleEvaluationRequest`：

```json
{
  "trace_id": "trace_20260604_0001",
  "task_id": "task_0001",
  "tenant_id": "tenant_demo",
  "business_id": "community_post",
  "policy_id": "default_policy",
  "policy_version": "v1",
  "modality": "text",
  "model_results": [
    {
      "model_name": "text_safety_v1",
      "model_version": "2026-06",
      "modality": "text",
      "labels": [
        {
          "label": "political_sensitive",
          "sub_label": "political_symbol",
          "score": 0.82,
          "normalized_score": 0.82
        }
      ],
      "evidence": [
        {
          "evidence_id": "ev_text_001",
          "type": "text_span",
          "content": "命中证据片段"
        }
      ],
      "latency_ms": 420,
      "status": "success",
      "error": null
    }
  ],
  "rag_evidence": []
}
```

字段约束：

- `trace_id` 必填，用于审计追踪。
- `tenant_id`、`business_id`、`policy_id`、`policy_version` 决定规则生效范围。
- `model_results` 必须使用标准化 `ModelResult` 结构。
- `rag_evidence` 可为空；当规则条件为 `rag_node` 时必须提供。

## 13. RuleResult

规则引擎返回 `RuleResult` 列表：

```json
{
  "rule_id": "rule_model_score_001",
  "version": "v1",
  "label": "political_sensitive",
  "condition_type": "model_score",
  "threshold": 0.8,
  "observed_value": 0.82,
  "matched": true,
  "action": "review",
  "evidence_refs": ["ev_text_001"],
  "reason": "模型 political_sensitive 分数 0.82 超过阈值 0.8"
}
```

字段约束：

- `condition_type` 对应 `keyword`、`regex`、`model_score`、`label_combo`、`rag_node` 或业务上下文条件。
- `observed_value` 是规则实际读取到的值。
- `matched` 表示是否命中。
- `evidence_refs` 引用 `ModelResult.evidence[].evidence_id` 或 Graph RAG 证据 ID。
- `reason` 面向前端展示和审计复盘，应说明命中原因。

## 14. model_score 映射规则

`model_score` 条件从 `ModelResult.labels` 读取模型分数：

1. 先按规则的 `label` 和可选 `sub_label` 匹配 `ModelResult.labels[]`。
2. 优先读取 `normalized_score`，如果缺失则读取 `score`。
3. 将读取到的值写入 `RuleResult.observed_value`。
4. 若 `observed_value >= PolicyRule.threshold`，则 `matched = true`。
5. 若多个模型返回同一标签，默认取最高 `normalized_score`；后续实现可按策略配置改为加权平均或指定模型优先。

示例：

```json
{
  "condition": {
    "type": "model_score",
    "label": "political_sensitive",
    "sub_label": "political_symbol"
  },
  "threshold": 0.8,
  "action": "review"
}
```

当 `ModelResult.labels[].normalized_score` 为 `0.82` 时，该规则命中并返回 `review`。

## 15. 错误处理

规则引擎错误使用统一错误对象：

```json
{
  "code": "RULE_INPUT_MISSING_MODEL_RESULT",
  "message": "model_score 规则缺少标准化模型结果",
  "retryable": true
}
```

建议错误码：

- `RULE_POLICY_NOT_FOUND`：策略不存在，不可重试。
- `RULE_VERSION_NOT_FOUND`：策略版本不存在，不可重试。
- `RULE_INPUT_MISSING_MODEL_RESULT`：缺少模型结果，可重试或等待异步补算。
- `RULE_INPUT_MISSING_RAG_EVIDENCE`：`rag_node` 规则缺少 RAG 证据，可重试。
- `RULE_EVALUATION_TIMEOUT`：规则执行超时，可重试。
