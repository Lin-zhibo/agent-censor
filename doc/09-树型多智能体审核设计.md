# 09-树型多智能体审核设计

## 1. 文档口径

本文档记录当前已经对齐的智能体设计结论。内容为目标设计和后续实现约定，不表示当前代码已经完成树型多智能体编排。

规则文件入口位于 `config/settings.json` 的 `rules` 字段。当前 `rules` 下包含 `security` 和 `ecosystem` 两个配置子树，分别映射到智能体设计中的 `SECURITY` 和 `ECOSYSTEM` 0 级标签。

后续正式规则文件预计包含以下字段：

```text
一级标签 - 二级标签 - 二级标签说明 - 三级标签（如有） - 三级标签说明（如有） - 关键词（如有）
```

本文档在此基础上补充 0 级标签和多智能体返回契约。

标签字段值统一使用大写英文，限制为 1-2 个 word；多个 word 使用下划线连接，例如 `TERROR`、`SUPPORT_SUICIDE`。不使用连字符或中文。常见前缀如 `ECO`、`AD`、`TERROR` 按 1 个 word 计算；中文名称和说明应放在规则文件、标签字典、`description` 或前端展示文案中。字段名仍保持 API 约定的 snake_case，例如 `security_labels`、`ecosystem_labels`。

## 2. 核心设计结论

已对齐的核心结论如下：

- 多智能体按标签树组织，而不是横向自由协作。
- 0 级标签是处置域，目前分为 `SECURITY` 和 `ECOSYSTEM`。
- `SECURITY` 命中后默认进入 `ban`、`reject` 或高优先级人工复核。
- `ECOSYSTEM` 命中后默认进入 `limit`、`pass_with_limit` 或普通复核。
- 每一级智能体都调用自己的子智能体，并对子智能体返回结果做综合后再返回给父智能体。
- 叶子智能体负责最小可执行标签，返回一个字典；`label` 为空字符串表示未命中，`label` 非空表示命中该叶子 label。
- 中间智能体返回一个字典，其中 `label` 是其子树下所有命中的叶子 label 数组，`reason` 是对子智能体返回原因的综合洞察。
- 顶级智能体返回 `security_labels`、`ecosystem_labels` 和 `reason`，其中 `reason` 是对两个 0 级子智能体返回原因的综合洞察。
- 父智能体不得修改、添加、删除收集到的子智能体标签，只能综合 `reason` 和补充审计过程。
- 最终处置由 0 级处置域、规则引擎结果和策略优先级共同决定。

## 3. 标签层级

建议将标签层级定义为：

```text
0级标签：处置域
  ├─ SECURITY（安全）
  │    ├─ 1级标签：PORN（色情低俗）
  │    ├─ 1级标签：BAD_VALUES（不良价值观）
  │    └─ 1级标签：ILLEGAL_BEHAVIOR（违法行为）
  │
  └─ ECOSYSTEM（生态）
       ├─ 1级标签：ECO_DIVERSION（导流）
       └─ 1级标签：ECO_INSULT（辱骂）
```

正式规则文件中的一级、二级、三级标签应挂载到 `config/settings.json` 的 `rules.security` 或 `rules.ecosystem` 下，并在智能体返回中规范化为 `SECURITY` 或 `ECOSYSTEM` 这两个 0 级处置域。0 级标签不只是分类字段，还决定默认处置动作和动作优先级。

## 4. 智能体树结构

建议智能体树与标签树保持一致：

```text
RootAgent
  ├─ SecurityAgent
  │    ├─ PornAgent
  │    │    ├─ PornServiceLeafAgent
  │    │    └─ PornBehaviorLeafAgent
  │    ├─ BadValuesAgent
  │    └─ IllegalAgent
  │
  └─ EcosystemAgent
       ├─ DiversionAgent
       └─ InsultAgent
```

说明：

- `RootAgent` 负责最终汇总，不直接执行具体审核规则。
- `SecurityAgent` 汇总安全域下所有叶子标签。
- `EcosystemAgent` 汇总生态域下所有叶子标签。
- 中间标签智能体负责子树调度、证据归并，并对子智能体返回的 `reason` 做综合洞察。
- 叶子智能体负责最小标签的规则执行准备和结果返回。

## 5. 执行方向

树型多智能体采用“自顶向下分发，自底向上汇总”的流程：

```text
1. RootAgent 接收审核上下文
2. RootAgent 将任务分发给 SecurityAgent 和 EcosystemAgent
3. 每个中间智能体继续分发给自己的子智能体
4. 叶子智能体执行最小标签判断
5. 叶子智能体返回 LeafLabelHit
6. 中间智能体聚合子树下所有 `label` 非空的结果，返回包含 `label` 数组和 `reason` 的字典
7. RootAgent 原样汇总 security_labels 和 ecosystem_labels，并综合子智能体 reason
8. 根据策略优先级生成最终处置建议
```

叶子智能体可以调用模型、规则、Graph RAG 等工具，但最终规则命中判断仍应由规则引擎完成，避免出现“智能体主观判断违规”的不可审计问题。

## 6. 返回契约

### 6.1 叶子智能体返回

叶子智能体返回一个字典。`label` 为空字符串表示未命中审核规则；`label` 非空表示命中对应叶子标签。

```json
{
  "label": "",
  "reason": "未检出问题"
}
```

命中时：

```json
{
  "label": "SEX_BEHAVIOR",
  "reason": "命中性行为相关规则，模型分数超过策略阈值。"
}
```

### 6.2 中间智能体返回

中间智能体返回一个字典。`label` 是其子树下所有命中的叶子 label 数组；`reason` 是对子智能体返回 `reason` 的综合洞察。

```json
{
  "label": ["SEX_SERVICE", "SEX_BEHAVIOR"],
  "reason": "REASONS"
}
```

未命中时：

```json
{
  "label": [],
  "reason": "未检出问题"
}
```

中间智能体可以对结果进行：

- 按子智能体返回顺序原样收集 `label`。
- 证据归并。
- insight 各个子智能体返回的 `reason`，形成本级 `reason`。

父智能体不得修改、添加、删除收集到的子智能体标签。具体包括：不改名、不去重、不排序、不新增、不删除。父智能体只能基于子智能体的 `reason` 形成自己的 `reason`。

### 6.3 顶级智能体返回

顶级智能体返回两个标签数组和一个综合解释。`security_labels` 与 `ecosystem_labels` 必须来自两个 0 级子智能体的返回结果，RootAgent 不得修改、添加、删除其中的标签；`reason` 是对各个子智能体 `reason` 的综合洞察。

```json
{
  "security_labels": ["LABEL1"],
  "ecosystem_labels": ["LABEL2"],
  "reason": ""
}
```

## 7. LeafLabelHit 字段

建议统一叶子命中对象为 `LeafLabelHit`：

| 字段 | 含义 |
| --- | --- |
| `label` | 最小命中标签名称；未命中时为空字符串 |
| `reason` | 面向前端和审计的命中解释 |
| `needs_review` | 可选，表示结果存在冲突或置信度不足，需要人工复核 |

## 8. 处置优先级

最终动作建议按以下优先级合并：

```text
SECURITY ban / reject
  > SECURITY review
  > ECOSYSTEM limit / pass_with_limit
  > ECOSYSTEM review
  > pass
```

推导规则建议：

```text
如果 security_labels 中存在 ban 或 reject：
    final_decision = reject
    suggested_action = ban
否则如果 security_labels 非空：
    final_decision = review
    suggested_action = manual_review
否则如果 ecosystem_labels 中存在 limit 或 pass_with_limit：
    final_decision = pass_with_limit
    suggested_action = limit
否则如果 ecosystem_labels 非空：
    final_decision = review
    suggested_action = manual_review
否则：
    final_decision = pass
    suggested_action = pass
```

如果同一内容同时命中安全域和生态域，安全域优先。

## 9. 规则文件映射

规则文件从 `config/settings.json` 的 `rules` 字段读取。建议读取后先归一化为标签树节点：

```json
{
  "node_id": "SECURITY.PORN.SEX_BEHAVIOR",
  "level": 3,
  "label": "SEX_BEHAVIOR",
  "description": "出现性行为相关内容",
  "parent_id": "SECURITY.PORN",
  "keywords": ["关键词1", "关键词2"],
  "rule_ids": ["RULE_SECURITY_PORN_BEHAVIOR_001"],
  "default_action": "ban",
  "action_priority": 100
}
```

映射原则：

- `rules.security` 映射为 `SECURITY` 处置域，`rules.ecosystem` 映射为 `ECOSYSTEM` 处置域。
- 0 级处置域由 `config/settings.json` 的 `rules` 字段和智能体树结构推导，不需要叶子智能体重复返回。
- 嵌套配置键表示标签层级，`name` 可作为展示名，`introduction` 可作为标签说明。
- 一级、二级、三级标签保持原始规则文件语义。
- `关键词` 可转换为 `keyword` 类型规则。
- 叶子标签应关联 `rule_ids`，便于叶子智能体调用规则引擎。
- 每个叶子标签必须能追溯到规则、证据和最终动作。

## 10. 与现有内部接口的关系

树型多智能体设计不新增对外 API。后续实现仍复用已有内部接口：

- `POST /internal/v1/models/route`：选择模型。
- `POST /internal/v1/models/infer`：执行模型推理。
- `POST /internal/v1/rules/query`：查询规则。
- `POST /internal/v1/rules/evaluate`：执行规则。
- `POST /internal/v1/rag/search`：检索政策、案例、规则和样本证据。
- `GET /internal/v1/audit/traces/{trace_id}`：查询审计轨迹。

树型智能体主要改变的是编排方式和返回契约，不改变模型、规则、Graph RAG 的职责边界。

## 11. 审计与前端展示

前端可基于顶级智能体返回结果展示：

- `security_labels`：安全风险列表，默认红色高风险展示。
- `ecosystem_labels`：生态风险列表，默认黄色或橙色中风险展示。
- 完整标签路径：由父节点和规则文件路径推导，例如 `SECURITY -> PORN -> SEX_BEHAVIOR`，前端可同时展示中文路径“安全 -> 色情低俗 -> 性行为相关”。
- 命中规则：从规则引擎结果或审计轨迹中读取。
- 证据高亮：从规则引擎结果、模型结果、Graph RAG 证据或审计轨迹中读取。
- 处置解释：展示 `reason` 和最终 `suggested_action`。

审计轨迹应保存每一级智能体的输入、子结果、综合结果和最终返回，便于复盘“为什么该叶子标签命中”和“为什么最终 ban 或限流”。

## 12. 后续待细化点

后续实现前还需要继续细化：

- 正式规则文件如何标注 0 级处置域。
- 每个叶子智能体是否一一对应一个叶子标签，还是允许一个叶子智能体覆盖多个相近叶子标签。
- 父级智能体综合低置信子结果 `reason` 时的审计字段。
- 多模态内容同时命中多个叶子标签时的证据合并规则。
- 树型智能体执行时是全量遍历，还是基于候选标签做剪枝。
