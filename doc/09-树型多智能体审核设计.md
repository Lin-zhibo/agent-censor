# 09-树型多智能体审核设计

## 1. 文档口径

本文档记录当前已经对齐的智能体设计结论。内容为目标设计和后续实现约定，不表示当前代码已经完成树型多智能体编排。

规则文件入口位于 `config/settings.json` 的 `rules` 字段。当前 `rules` 下包含 `security` 和 `ecosystem` 两个配置子树，它们只是领域分组，用于把命中结果归入 `security_labels` 或 `ecosystem_labels`，不是 0 级智能体。

后续正式规则文件预计包含以下字段：

```text
一级标签 - 二级标签 - 二级标签说明 - 三级标签（如有） - 三级标签说明（如有） - 关键词（如有）
```

本文档在此基础上补充 0 级顶级智能体、多智能体提示词和返回契约。

标签字段值统一使用大写英文，限制为 1-2 个 word；多个 word 使用下划线连接，例如 `TERROR`、`SUPPORT_SUICIDE`。不使用连字符或中文。常见前缀如 `ECO`、`AD`、`TERROR` 按 1 个 word 计算；中文名称和说明应放在规则文件、标签字典、`description` 或前端展示文案中。字段名仍保持 API 约定的 snake_case，例如 `security_labels`、`ecosystem_labels`。

## 2. 核心设计结论

已对齐的核心结论如下：

- 多智能体按标签树组织，而不是横向自由协作。
- 0 级智能体是 `RootAgent`，负责看到全部一级标签并调度一级子智能体。
- `security` 和 `ecosystem` 只是领域分组，不是智能体层级；领域分组用于决定结果写入 `security_labels` 还是 `ecosystem_labels`。
- `rules.security` 和 `rules.ecosystem` 下面的直接子节点才是一级标签，例如 `DISTORT_TRADITION`、`TERROR`、`ECO_DIVERSION`。
- 一级标签命中后，根据所属领域、Agent 动作建议和动作优先级进入 `ban`、`reject`、`limit`、`pass_with_limit` 或人工复核。
- 每一级智能体都调用自己的子智能体，并对子智能体返回结果做综合后再返回给父智能体。
- 所有非叶子智能体在提示词输入中必须看到其直接子节点的 `label`、`name`、`description` 和所属领域，否则无法正确选择或调度子智能体。
- 叶子智能体负责最小可执行标签，返回一个字典；`label` 为空字符串表示未命中，`label` 非空表示命中该叶子 label。
- 中间智能体返回一个字典，其中 `label` 是其子树下所有命中的叶子 label 数组，`reason` 是对子智能体返回原因的综合洞察。
- 顶级智能体返回 `security_labels`、`ecosystem_labels` 和 `reason`，其中 `reason` 是对全部一级子智能体返回原因的综合洞察。
- 父智能体不得修改、添加、删除收集到的子智能体标签，只能综合 `reason` 和补充审计过程。
- 最终处置由命中标签所属领域、Agent 命中标签、Agent 给出的动作建议和动作优先级共同决定。

## 3. 标签层级

建议将标签层级定义为：

```text
0级智能体：RootAgent（顶级调度与最终汇总）
  ├─ 领域分组：security（不是智能体层级）
  │    ├─ 1级标签：DISTORT_TRADITION（歪曲优秀传统）
  │    ├─ 1级标签：TERROR（传播恐怖主义）
  │    └─ 1级标签：ILLEGAL_BEHAVIOR（违法行为）
  │
  └─ 领域分组：ecosystem（不是智能体层级）
       ├─ 1级标签：ECO_DIVERSION（导流）
       └─ 1级标签：ECO_INSULT（辱骂）
```

正式规则文件中的一级、二级、三级标签应挂载到 `config/settings.json` 的 `rules.security` 或 `rules.ecosystem` 下。`rules.security` 和 `rules.ecosystem` 只决定领域归属，不额外生成领域汇总层智能体。顶级 `RootAgent` 应直接看到两个领域下全部一级标签的 `name`，并据此调度一级子智能体。

## 4. 智能体树结构

建议智能体树与标签树保持一致：

```text
RootAgent
  ├─ DistortTraditionAgent（domain=security，name=歪曲优秀传统）
  │    ├─ HistoryDistortLeafAgent（name=中国历史恶意传播）
  │    └─ CultureDistortLeafAgent（name=不当解读中国传统文化）
  ├─ TerrorAgent（domain=security，name=传播恐怖主义）
  ├─ IllegalBehaviorAgent（domain=security，name=违法行为）
  ├─ EcoDiversionAgent（domain=ecosystem，name=导流）
  └─ EcoInsultAgent（domain=ecosystem，name=辱骂）
```

说明：

- `RootAgent` 是 0 级智能体，负责看到全部一级标签的 `label`、`name`、`description` 和领域归属，并调度一级子智能体。
- `security` 和 `ecosystem` 不对应一层智能体，只是一级标签的领域属性。
- 中间标签智能体负责看到自己全部直接子节点的 `label`、`name`、`description`，完成子树调度、证据归并，并对子智能体返回的 `reason` 做综合洞察。
- 叶子智能体负责最小标签的审核判断准备和结果返回。

## 5. 执行方向

树型多智能体采用“自顶向下分发，自底向上汇总”的流程：

```text
1. RootAgent 接收审核上下文
2. RootAgent 读取全部一级标签的 label、name、description 和 domain
3. RootAgent 将任务分发给一级标签智能体
4. 每个中间智能体继续分发给自己的子智能体
5. 叶子智能体执行最小标签判断
6. 叶子智能体返回 LeafLabelHit
7. 中间智能体聚合子树下所有 `label` 非空的结果，返回包含 `label` 数组和 `reason` 的字典
8. RootAgent 按一级标签所属 domain 原样汇总 security_labels 和 ecosystem_labels，并综合子智能体 reason
9. 根据 Agent 返回的动作优先级生成最终处置建议
```

叶子智能体是最小标签的审核裁决主体，可以调用模型、标签字典、Graph RAG、历史审计等外部工具补充证据，但外部工具只提供输入和证据，不替代 Agent 的审核结论。

## 6. 三层智能体提示词模板设计

三层智能体提示词需要和标签树职责保持一致：RootAgent 只做顶级调度和最终处置汇总，中间智能体只做子树调度与原因综合，叶子智能体只判断一个最小可执行标签。所有提示词都必须强调“标签由叶子 Agent 独立审核产生，父智能体不得主观增删标签”。

### 6.1 通用提示词约束

所有层级智能体共用以下约束：

```text
你是内容安全审核系统中的一个树型智能体节点。
你必须严格遵守当前节点职责，不得越级判断其它节点标签。
标签字段必须使用规则树中的英文大写 label，不能输出中文标签、同义标签或新标签。
父级智能体不得修改、添加、删除、去重或排序子智能体返回的 label。
如果当前 Agent 没有足够证据支撑命中，必须返回未命中或人工复核结果。
输出必须是合法 JSON，不要输出 Markdown、解释性前缀或额外自然语言。
reason 必须面向审计，说明依据、命中原因、冲突或未命中原因。
涉及命中判断时，必须由当前 Agent 基于标签定义、审核上下文和证据独立给出结论；外部工具结果只能作为证据输入，不能覆盖 Agent 结论。
```

建议每次调用都注入以下通用变量：

```json
{
  "trace_id": "{trace_id}",
  "tenant_id": "{tenant_id}",
  "business_id": "{business_id}",
  "policy_id": "{policy_id}",
  "policy_version": "{policy_version}",
  "modality": "{modality}",
  "content": "{content}",
  "content_features": "{content_features}",
  "labels_requested": "{labels_requested}",
  "node": {
    "node_id": "{node_id}",
    "level": "{level}",
    "domain": "{security|ecosystem}",
    "label": "{label}",
    "name": "{name}",
    "description": "{description}",
    "path": "{path}"
  },
  "child_nodes": [
    {
      "node_id": "{child_node_id}",
      "level": "{child_level}",
      "domain": "{security|ecosystem}",
      "label": "{child_label}",
      "name": "{child_name}",
      "description": "{child_description}",
      "is_leaf": false
    }
  ]
}
```

`child_nodes` 只提供给非叶子智能体。叶子智能体没有子节点，不需要该字段。任何非叶子智能体调用子智能体前，都必须先看到这些直接子节点的 `name`，不能只看到英文 `label`。

### 6.2 RootAgent 提示词模板

RootAgent 是 0 级智能体，负责看到 `rules.security` 和 `rules.ecosystem` 下全部一级标签的 `name`，并据此调度一级子智能体。`security` 和 `ecosystem` 是领域字段，不是 RootAgent 的两个子智能体。

系统提示词模板：

```text
你是 RootAgent，负责内容安全审核任务的顶级编排。
你的职责：
1. 接收审核上下文。
2. 读取全部一级标签子节点的 label、name、description 和 domain。
3. 根据一级标签 name 和 description 调度一级子智能体。
4. 原样收集一级子智能体返回的命中 label。
5. 按一级子智能体所属 domain 将命中 label 写入 security_labels 或 ecosystem_labels。
6. 综合全部一级子智能体的 reason，生成顶级 reason。
7. 根据领域、动作建议和动作优先级生成 final_decision 和 suggested_action。

硬性约束：
- 不得直接新增、删除、改名、去重或排序任何子智能体返回的 label。
- 不得把 security 或 ecosystem 当作子智能体调用；它们只是领域字段。
- security_labels 只能收集 domain=security 的一级子智能体返回 label。
- ecosystem_labels 只能收集 domain=ecosystem 的一级子智能体返回 label。
- 如果全部一级子智能体均未命中，reason 返回“未检出问题”。
- 如果 security 领域和 ecosystem 领域同时命中，最终处置以 security 领域优先。
- 调度一级子智能体前必须能看到全部一级标签的 name。
- 输出必须是合法 JSON。
```

用户提示词模板：

```text
请基于以下审核上下文、全部一级标签子节点和一级子智能体结果，生成 RootAgent 返回结果。

审核上下文：
{audit_context_json}

全部一级标签子节点，RootAgent 必须能看到 name：
{first_level_children_json}

一级子智能体结果，顺序不可改变：
{first_level_results_json}

处置优先级：
security ban/reject > security review > ecosystem limit/pass_with_limit > ecosystem review > pass

请只输出如下 JSON：
{
  "security_labels": [],
  "ecosystem_labels": [],
  "reason": "",
  "final_decision": "pass | review | reject | pass_with_limit",
  "suggested_action": "pass | manual_review | ban | limit"
}
```

RootAgent 输出要求：

- `security_labels`：按一级子智能体所属 `domain=security` 收集其返回的非空 `label`。
- `ecosystem_labels`：按一级子智能体所属 `domain=ecosystem` 收集其返回的非空 `label`。
- `reason`：合并非“未检出问题”的子级原因；都未命中时返回“未检出问题”。
- `final_decision` 和 `suggested_action`：按第 10 节处置优先级推导。

### 6.3 中间智能体提示词模板

中间智能体是任意非叶子标签智能体，例如 `DistortTraditionAgent`、`TerrorAgent`、`IllegalBehaviorAgent`、`EcoDiversionAgent`。中间智能体负责看到直接子节点的 `name`，调用子智能体、汇总子树标签和综合原因。

系统提示词模板：

```text
你是树型内容安全审核系统中的中间智能体。
你只负责当前节点及其子树的调度和汇总，不直接创造命中标签。

你的职责：
1. 根据当前节点信息理解本子树审核范围。
2. 读取当前节点全部直接子节点的 label、name、description 和 domain。
3. 根据子节点 name 和 description 调用当前节点的全部子智能体，或读取已经提供的 child_results。
4. 按子智能体返回顺序收集所有非空 label。
5. 综合子智能体 reason，生成当前节点 reason。
6. 返回当前子树命中的叶子 label 数组。

硬性约束：
- label 数组只能来自子智能体返回结果。
- 不得新增、删除、改名、去重或排序子智能体返回的 label。
- 不得把当前中间节点 label 当作命中标签返回，除非当前节点本身就是叶子节点。
- 调用子智能体前必须看到当前节点的直接子节点 name。
- 子智能体均未命中时，label 返回空数组，reason 返回“未检出问题”。
- 输出必须是合法 JSON。
```

用户提示词模板：

```text
请基于当前节点和子智能体返回结果，生成中间智能体返回结果。

当前节点：
{node_json}

当前节点的直接子节点，必须包含 name：
{child_nodes_json}

审核上下文：
{audit_context_json}

子智能体结果，顺序不可改变：
{child_results_json}

请只输出如下 JSON：
{
  "label": [],
  "reason": ""
}
```

中间智能体输出要求：

- `label`：遍历 `child_results`，按原顺序收集所有非空叶子 label；如果某个子结果的 `label` 是数组，则按数组原顺序展开。
- `reason`：综合所有非“未检出问题”的子原因，保留关键依据、标签定义、证据和冲突信息；没有命中时返回“未检出问题”。
- 中间智能体只能形成更清晰的原因摘要，不能改变审核结论。

### 6.4 叶子智能体提示词模板

叶子智能体负责最小标签判断。每个叶子智能体只对应一个叶子 `label`，例如 `SEX_SERVICE`、`FRAUD`、`PRIVATE_CHAT`。叶子智能体可以调用模型、标签字典、Graph RAG 和审计工具，但最终命中由叶子 Agent 自主给出。

系统提示词模板：

```text
你是内容安全审核系统中的叶子智能体，只负责判断一个最小标签是否命中。
当前叶子标签由 node.label 指定，你不得判断或输出其它标签。

你的职责：
1. 读取当前叶子标签的 label、name、description、keywords、default_action 和 action_priority。
2. 必要时调用 model_route 选择模型。
3. 必要时调用 run_model_inference 获取模型候选标签、分数和证据。
4. 可调用 graph_rag_search、audit_trace_lookup 或其它证据工具补充政策、案例、历史样本和上下文证据。
5. 根据当前标签定义、审核上下文、模型候选结果和证据材料，独立判断当前叶子标签是否命中。
6. 在 reason 中说明命中或未命中的关键依据。
7. 输出 LeafLabelHit JSON。

硬性约束：
- 只能输出当前 node.label 或空字符串。
- 命中时 label 必须等于当前 node.label。
- 未命中、证据不足或当前标签不适用于该内容时 label 必须为空字符串。
- 模型结果、RAG 结果和历史审计结果都是证据输入，不能强制覆盖 Agent 的独立审核结论。
- 外部工具结果之间存在冲突时，由 Agent 在 reason 中说明冲突、取舍和最终判断依据。
- 工具失败但风险无法排除时，可返回 needs_review=true 并说明需要人工复核。
- 输出必须是合法 JSON。
```

用户提示词模板：

```text
请判断当前内容是否命中指定叶子标签。

当前叶子节点：
{node_json}

审核上下文：
{audit_context_json}

模型路由结果：
{model_route_result_json}

模型推理结果：
{model_inference_result_json}

Graph RAG 证据：
{graph_rag_result_json}

历史审计或外部证据：
{external_evidence_json}

请只输出如下 JSON：
{
  "label": "",
  "reason": "未检出问题",
  "needs_review": false,
  "evidence": [],
  "action": "",
  "action_priority": 0
}
```

叶子智能体输出要求：

- 命中时：

```json
{
  "label": "{node.label}",
  "reason": "Agent 判定命中{node.display_name}：{evidence_summary}，处置建议为{action}。",
  "needs_review": false,
  "evidence": [
    {
      "type": "agent_evidence",
      "content": "{evidence_content}"
    }
  ],
  "action": "{action}",
  "action_priority": 100
}
```

- 未命中时：

```json
{
  "label": "",
  "reason": "未检出问题",
  "needs_review": false,
  "evidence": [],
  "action": "{default_action}",
  "action_priority": 0
}
```

- 需人工复核时：

```json
{
  "label": "",
  "reason": "外部证据提示存在风险，但 Agent 证据不足以直接命中，建议人工复核。",
  "needs_review": true,
  "evidence": [
    {
      "type": "model",
      "content": "{model_evidence_summary}"
    }
  ],
  "action": "manual_review",
  "action_priority": 0
}
```

## 7. 返回契约

### 7.1 叶子智能体返回

叶子智能体返回一个字典。`label` 为空字符串表示未命中当前审核标签；`label` 非空表示命中对应叶子标签。

```json
{
  "label": "",
  "reason": "未检出问题"
}
```

命中时：

```json
{
  "label": "HISTORY_DISTORT",
  "reason": "Agent 判定命中中国历史恶意传播标签，依据为文本语义、标签说明和证据片段。"
}
```

### 7.2 中间智能体返回

中间智能体返回一个字典。`label` 是其子树下所有命中的叶子 label 数组；`reason` 是对子智能体返回 `reason` 的综合洞察。

```json
{
  "label": ["HISTORY_DISTORT", "CULTURE_DISTORT"],
  "reason": "REASONS"
}
```

未命中时：

```json
{
  "label": [],
  "reason": "未检出问题"
}
```

中间智能体可以对结果进行：

- 按子智能体返回顺序原样收集 `label`。
- 证据归并。
- insight 各个子智能体返回的 `reason`，形成本级 `reason`。

父智能体不得修改、添加、删除收集到的子智能体标签。具体包括：不改名、不去重、不排序、不新增、不删除。父智能体只能基于子智能体的 `reason` 形成自己的 `reason`。

### 7.3 顶级智能体返回

顶级智能体返回两个标签数组和一个综合解释。`security_labels` 与 `ecosystem_labels` 必须来自全部一级子智能体的返回结果，并按一级子智能体所属 `domain` 归类。RootAgent 不得修改、添加、删除其中的标签；`reason` 是对各个一级子智能体 `reason` 的综合洞察。

```json
{
  "security_labels": ["LABEL1"],
  "ecosystem_labels": ["LABEL2"],
  "reason": ""
}
```

## 8. 调用提示词模板

数据流遵循 father → call → son → return → father。第 7 节定义了 son → father 的返回契约（自底向上），本节定义 father → son 的调用提示词模板（自顶向下）。

调用提示词由父智能体为每个直接子节点构造，作为子智能体的用户提示词注入。子智能体自身的系统提示词见第 6 节。叶子节点无子节点，不设调用模板。

调用提示词中引用的标签字段来自 `config/settings.json` 中 `rules.security` 或 `rules.ecosystem` 下的节点定义，字段含义如下：

| 提示词字段 | settings.json 来源 | 说明 |
| --- | --- | --- |
| `label` | `label` | 英文大写标签（如 `DISTORT_TRADITION`） |
| `name` | `name` | 中文标签名称（如 "歪曲优秀传统"） |
| `introduction` | `introduction` | 标签说明，含法律依据、命中条件和不命中条件 |

### 8.1 RootAgent 调用一级子智能体

RootAgent 为 `rules.security` 和 `rules.ecosystem` 下每个一级子智能体分别构造调用提示词。`security` 和 `ecosystem` 是领域分组，不对应子智能体调用。

```text
请基于以下审核上下文，完成当前节点及其子树的审核调度和结果汇总。

审核上下文：
{audit_context_json}

当前节点对应 config/settings.json 中 rules.{security|ecosystem} 下的以下标签定义：
{
  "label": "{label}",
  "name": "{name}",
  "introduction": "{introduction}"
}

当前节点的直接子节点（供继续向下调度）：
{child_nodes_json}

每个子节点的调用提示词中会包含其 label、name、introduction 字段。

请按 child_nodes 顺序逐个调度子智能体，收集全部非空 label，综合子智能体 reason 后返回。
必须输出合法 JSON：
{
  "label": [],
  "reason": ""
}
```

### 8.2 中间智能体调用子中间智能体

中间智能体为非叶子子节点构造调用提示词：

```text
请基于以下审核上下文，完成当前节点及其子树的审核调度和汇总。

审核上下文：
{audit_context_json}

当前节点对应 config/settings.json 中的以下标签定义：
{
  "label": "{label}",
  "name": "{name}",
  "introduction": "{introduction}"
}

当前节点的直接子节点：
{child_nodes_json}

请按顺序调度子智能体，收集全部非空 label，综合 reason 后返回。
必须输出合法 JSON：
{
  "label": [],
  "reason": ""
}
```

### 8.3 中间智能体调用叶子智能体

中间智能体为叶子子节点构造调用提示词：

```text
请判断当前内容是否命中以下叶子标签。可调用模型、RAG 等外部工具获取证据，但须独立给出审核结论。

审核上下文：
{audit_context_json}

当前叶子节点对应 config/settings.json 中的以下标签定义：
{
  "label": "{label}",
  "name": "{name}",
  "introduction": "{introduction}"
}

只输出如下 JSON，不含 null 值：
{
  "label": "",
  "reason": "未检出问题",
  "needs_review": false,
  "evidence": []
}

硬性约束：
- 只能判断当前叶子标签，不得输出其他标签。
- 命中时 label 必须等于当前叶子 label；未命中时 label 为空字符串。
- 模型、RAG 结果为证据输入，不能覆盖 Agent 独立审核结论。
```

## 9. LeafLabelHit 字段

建议统一叶子命中对象为 `LeafLabelHit`：

| 字段 | 含义 |
| --- | --- |
| `label` | 最小命中标签名称；未命中时为空字符串 |
| `reason` | 面向前端和审计的命中解释 |
| `needs_review` | 可选，表示结果存在冲突或置信度不足，需要人工复核 |

## 10. 处置优先级

最终动作建议按以下优先级合并：

```text
security 领域 ban / reject
  > security 领域 review
  > ecosystem 领域 limit / pass_with_limit
  > ecosystem 领域 review
  > pass
```

推导逻辑建议：

```text
如果 security_labels 中存在 ban 或 reject：
    final_decision = reject
    suggested_action = ban
否则如果 security_labels 非空：
    final_decision = review
    suggested_action = manual_review
否则如果 ecosystem_labels 中存在 limit 或 pass_with_limit：
    final_decision = pass_with_limit
    suggested_action = limit
否则如果 ecosystem_labels 非空：
    final_decision = review
    suggested_action = manual_review
否则：
    final_decision = pass
    suggested_action = pass
```

如果同一内容同时命中安全域和生态域，安全域优先。

## 11. Agent 审核模块接口

树型多智能体审核是一个独立模块，对外提供统一审核接口。调用方只提交审核上下文，接收 Agent 审核结果；调用方不需要直接感知树内 RootAgent、中间智能体和叶子智能体的执行细节。

建议对外接口：

- `POST /internal/v1/agents/moderate`：提交单条内容审核任务，返回 `security_labels`、`ecosystem_labels`、`reason`、`final_decision` 和 `suggested_action`。
- `POST /internal/v1/agents/moderate:batch`：提交批量审核任务，返回逐条 Agent 审核结果。
- `GET /internal/v1/agents/traces/{trace_id}`：查询树型 Agent 审核轨迹。
- `POST /internal/v1/agents/preview`：基于临时标签说明、动作优先级或审核配置预览 Agent 审核结果，不写入正式审核结论。

Agent 模块内部可以按需调用模型、Graph RAG、标签字典、历史审计或其它证据服务，但这些服务只作为 Agent 的输入依赖。对外契约上，审核结论由 Agent 模块产出。

## 12. 审计与前端展示

前端可基于顶级智能体返回结果展示：

- `security_labels`：安全风险列表，默认红色高风险展示。
- `ecosystem_labels`：生态风险列表，默认黄色或橙色中风险展示。
- 完整标签路径：由父节点和规则文件路径推导，例如 `security -> DISTORT_TRADITION -> HISTORY_DISTORT`，前端可同时展示中文路径“安全领域 -> 歪曲优秀传统 -> 中国历史恶意传播”。
- 命中依据：从 Agent 返回的 `reason`、`evidence` 或审计轨迹中读取。
- 证据高亮：从 Agent 证据、模型候选结果、Graph RAG 证据或审计轨迹中读取。
- 处置解释：展示 `reason` 和最终 `suggested_action`。

审计轨迹应保存每一级智能体的输入、子结果、综合结果和最终返回，便于复盘“为什么该叶子标签命中”和“为什么最终 ban 或限流”。

## 13. 后续待细化点

后续实现前还需要继续细化：

- 正式规则文件如何从 `security` / `ecosystem` 领域分组注入 `domain` 字段。
- 每个叶子智能体是否一一对应一个叶子标签，还是允许一个叶子智能体覆盖多个相近叶子标签。
- 父级智能体综合低置信子结果 `reason` 时的审计字段。
- 多模态内容同时命中多个叶子标签时的证据合并逻辑。
- 树型智能体执行时是全量遍历，还是基于候选标签做剪枝。
